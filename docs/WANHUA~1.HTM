萬華協力聯盟 · 待用券管理系統 — 流程圖 + Story Map v10 資料模型確定

Planning v1 · 4 個角色 · 6 階段 · 兩個 Release · 含資料模型 + 金流方向 (立心 → NGO 代表 → 店家)

1. 已對齊的決定 (v2 更新)

  1. 券單價：每張券固定 NT$ 100
  2. 角色合併：NGO 與 NGO 代表合併為「NGO 代表」（一個人同時填單位需求、分券給個案、月底跑店補款）
  3. 補款金流：由 NGO 代表現場帶現金補給店家（v1 不需要金流模組）
  4. 代表–店家關係：一個 NGO 代表可能負責多家店 → 月底回收 UI 需支援多店切換
  5. 未用券處理：自動折抵下月配額（不作廢、不重發）
  6. 個案資料：個案沒有帳號，只是 NGO 系統中的一筆資料
  7. 店家入口：店家完全被動，全部由 NGO 代表代為操作（v1 不做店家 portal）
  8. 券流水號設計：立心統一發放，但流水號沒有任何編碼邏輯（例如 NO. 27031 純流水）→ 系統無法從券號直接推斷店家，完全依賴 NGO 代表上傳的對應表。
  9. 系統不追蹤庫存：NGO 代表手上會有跨月累積的實體券（不回收），但系統不管庫存狀態。系統只紀錄「個案領了哪些券」與「店家收到哪些券」，未上傳的券就不存在系統中。
  10. 結算金流方向：NGO 代表現場用現金補給店家 → 店家開收據給 NGO 代表 → NGO 代表拿收據向立心請款。
  11. 立心每月一次性付款：立心一個月只匯一次款，把「下月預付各店家」+「上月還 NGO 代表現金」合併為一筆匯款，發生在月底結算階段 (STEP 5)。
  12. 立心發券給 NGO 代表是純線下動作：不進系統。系統第一次「看見」券是在 NGO 代表發券給個案時的第一次上傳。
  13. 兩次上傳設計（核心架構）：
    ① 第一次上傳（發券前）：NGO 代表上傳「流水號 → 屬於哪家店 → 發給哪個個案 → 發幾張」
    ② 第二次上傳（月底收券）：用相機掃描店家收到的所有券，系統自動比對第一次上傳的紀錄 → 標示他店券 → 算補款 → 也能反查哪些個案使用了券
  14. 個案資料：NGO 代表先上傳個案名單到系統建檔，發券時用查詢的方式選個案。系統不需管「分券給個案的邏輯」（每月幾張、誰可以拿等等都不管）。
  15. NGO 代表–店家關係：每月固定，不輪替。
  16. 跨 NGO 用券：v1 假設不會發生（NGO A 的個案不會持 NGO B 領到的券）。
  17. 機構用印處系統不管：誰蓋章、什麼時機蓋都不追蹤。
  18. 個案不需要身分驗證，也不會發生「店家用印處未蓋章」的容錯情境（一定會蓋）。
  19. 個案資訊欄位：姓名 + 所屬 NGO + 身分證字號（需注意資安儲存）。
  20. 立心月度結算單可下載 CSV，用於對帳與審核 NGO 代表的請款收據。
  21. 需求視角已確認：NGO 想知道「我的個案使用了哪些券」；立心想看「全部待用券的使用情況」（個別 NGO 視角 + 全域視角都要做）。
  22. 券月份戳的時機：未發出的券沒有月份（空白），月份是 NGO 代表發放給個案時才寫上去 → 系統內所有上傳的券都是當月發、當月用，無「跨月剩券」的問題。
  23. 身分證字號儲存方式：明碼（v1 採此方案）。備註：未來若有資安考量可改為加密 / 後 4 碼遮罩。
  24. 第二次上傳相機只讀流水號：不識別店家章圖像。NGO 代表先手動選自己現在在哪家店，相機只負責讀流水號數字。
  25. OCR 技術選型：使用手機原生 OCR（iOS Vision Framework / Android ML Kit），不需要 AI API。離線運作、免費、適合「統一印刷紅色 5 位數 + 流水號印兩次」的情境，可雙位置交叉比對提升準確率。未來如實測準確率 < 90%，再加雲端 API（Google Vision / Claude / GPT-4o Vision）作為 fallback。
  26. 資料建立權限：個案 + 店家 由 NGO 代表自行建立（在自己負責的範圍內）；NGO 代表帳號由 立心建立（不開放自助註冊）。
  27. 金流方向（v10 修正）：立心 → NGO 代表 → 店家（單一鏈）。立心不直接付店家，所有給店家的錢（下月預付 + 他店券補款）統一給 NGO 代表，NGO 代表帶現金分發給負責的店家。
  28. 結算單算法：立心每月給每位 NGO 代表的金額 = Σ (該代表負責店家的下月預付 100×張數) + Σ (該代表負責店家的本月他店券 100×張數)。
  29. 收據上傳：NGO 代表交付現金給店家後拍照上傳（含收據照片 + 收到日期）；立心可檢視所有收據用於對帳，但不負責上傳。
  30. 立心 = admin + 自家 NGO 代表 雙重身份：立心基金會本身也是參與計畫的 NGO，有自己負責的店家與個案。所以立心人員擁有 admin 全域權限 (建帳號、審核結算、看全 NGO 資料) ＋ 對自家 NGO 的 NGO 代表功能 (建個案/店家、發券、月底回收、上傳收據)。系統上他們是 role='lixin' 但 ngo_id = 立心 NGO 的 user。
  31. 發佈方式：PWA (漸進式網頁 App)，不上 App Store / Play Store。單一 Next.js 專案，NGO 代表用手機瀏覽器「加到主畫面」就能像 native app 一樣用。OCR 用 Tesseract.js (browser-side)。
    好處：省 Apple Dev fee ($99/year)、不用過審核、改版即時生效、單一 codebase。

1.5 實體券資訊 (從照片觀察)

  • 名稱：「待用餐券」(Voucher) NT$ 100，標語「牽起社區關係」
  • 流水號：NO. 27031 樣式（5 位純數字，無店家編碼），同號出現兩次（正面右上 + 左側備註欄上方）
  • 欄位：月份欄 (右上)、店家用印處 (中間左圓)、機構用印處 (中間右圓)、備註欄 (左側)
  • 條件：「當月使用、逾期無效」、「未蓋店家與機構章者無效」
  • 啟示：實體券已印好無法重印，所有「店家 / 個案」資訊都得透過系統紀錄
兩次上傳的心智模型
立心拿到實體券 → 派發給 NGO 代表（線下、不入系統）→ NGO 代表決定「這張券要發給哪家店的個案」並蓋機構章 → ① 上傳對應表（流水號 → 店家 → 個案）→ 個案拿券到店家 → 店家蓋章兌換 → 月底 NGO 代表去店家收券 → ② 相機掃描上傳 → 系統比對：店家實際收到的券 vs. 第一次上傳的對應表 → 自動算補款 + 反查個案使用紀錄。

系統不追蹤實體券庫存。未上傳的券就是 NGO 代表手上的紙片，系統不知道也不需要知道。

5. Release 切片 · 每個 Release 走得通整條流程

Release 1 — MVP：建立兩次上傳骨架

R1 重點是做完整的「兩次上傳 + 一次性結算」骨架。第一次上傳（發券前指派）在 R1 就要完整，因為這是補款計算的 source of truth。第二次上傳（月底）R1 先做手動輸入版 + 自動比對 + 自動算補款 + 反查個案。相機掃描留到 R2。

結算流程完整：NGO 代表現場給現金 + 拍店家收據 → 系統產生立心月度結算單（下月預付各店家 ＋ 還款給 NGO 代表合併） → 立心一次匯款。

使用者可以做到：完整跑完一個月循環、補款金額不會算錯、能反查每張券給了誰、立心每月只需做一次匯款動作。

Release 2 — 相機掃描 + 體驗優化

導入相機掃描（第一次發券前 + 第二次月底回收都可用），把手 key 流水號的時間大幅降低。立心開始能用儀表板看每月趨勢、識別跨店問題店家、收據自動歸檔。

使用者可以做到：跑店時間大幅縮短；立心用資料看每月趨勢；NGO 代表用相機快速掃描券號。
