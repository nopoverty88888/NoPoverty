# 待用券管理系統 — 功能規格

## 1. 名詞與角色

- **立心 (Lixin Foundation)** — 待用券計畫統籌者，同時<b>本身也是一個 NGO</b>（有自己負責的店家與個案）。所以立心人員的帳號是<b>雙重身份</b>：admin 全域權限（建帳號、審核結算、看全 NGO 資料）<b>＋</b> 對自家 NGO 的 NGO 代表功能（建個案/店家、發券、月底回收、上傳收據）。
- **NGO** — 參與待用券計畫的單位 (e.g. 立心、勵馨等)。立心本身就是 NGOs 列表中的一筆。
- **NGO 代表** — 一個 NGO 內的工作人員 (一個 NGO 通常一位)。負責：填單位需求、管理個案、發券給個案、月底跑店家收券補款、上傳收據。一位 NGO 代表負責多家店家。<b>立心人員雖然 role=lixin，但同時擁有 NGO 代表功能。</b>
- **店家** — 接受待用券兌換餐點的合作餐廳。系統內被動，不登入。
- **個案** — 持券到店家兌換餐點的弱勢者。系統內被動，不登入，只是 NGO 代表系統中的一筆資料。
- **待用券 (Voucher)** — 紙本實體券，固定 NT$100。流水號無編碼邏輯 (5 位純數字)。

## 2. 月度循環 (六個階段)

1. **需求統計** — NGO 代表填寫單位對各店家本月需要的券數，立心彙整。
2. **券分配給個案 (第一次上傳)** — NGO 代表把實體券發給個案，並上傳「流水號 → 屬於哪家店 → 發給哪個個案」。
3. **個案兌換** — 個案持券到店家，店家在「店家用印處」蓋章後出餐。完全線下、不進系統。
4. **月底回收 (第二次上傳)** — NGO 代表到負責店家收券，用相機掃描流水號 (R2) 或手動輸入 (R1)，系統自動比對找出他店券、算出補款金額、反查使用個案。
5. **結算 + 月度一次性匯款** — 系統產生立心月度結算單（每位 NGO 代表的金額 = 下月預付各店家 + 本月他店券補款）。立心審核後一次匯款給 NGO 代表。NGO 代表帶現金分發給負責店家，並拍收據上傳。
6. **下月規劃** — NGO 代表看到個案使用紀錄、立心看到全域使用情況。

## 3. 核心架構：兩次上傳 + 一次匯款

### 第一次上傳 (STEP 2)
NGO 代表發券「之前」必須先在系統內紀錄「這張流水號要發給哪家店的哪位個案」。
這是補款計算的 source of truth。

### 第二次上傳 (STEP 4)
NGO 代表月底到店家收券，把收到的所有券逐張掃描 (R2) 或手 key (R1)。
系統自動比對：
- 比對 voucher_assignments 表 → 找出「此券原本指派給哪家店」
- 比對 NGO 代表現在所在的店 → 若兩者不同 → 標記為「他店券」
- 補款 = 他店券張數 × NT$100

### 一次匯款 (STEP 5)
立心一個月只匯一次款，全部發給 NGO 代表 (不是直接給店家)：
```
給 NGO 代表的金額
  = Σ (該代表負責店家的下月預付：100 × 張數)
  + Σ (該代表負責店家的本月他店券：100 × 張數)
```
NGO 代表收款後，帶現金分發給負責的店家。

## 4. 三項新功能（相對於目前的試算表）

1. **個案使用紀錄表** — 系統能反查每個個案使用了哪些券 (從 voucher_collections + voucher_assignments join)。
2. **流水號指名功能** — 第一次上傳就是這個。批量指派流水號給店家+個案。
3. **相機掃描功能** — 第二次上傳的 R2 強化，把月底登記時間從 1-2 小時降到 10 分鐘。OCR 用手機原生 (iOS Vision / Android ML Kit)，**不需要 AI API**。

## 5. Release 切片

### Release 1 — MVP：兩次上傳骨架
- 全部流程跑得通，第二次上傳是手 key 流水號版
- 立心可下載月度結算單 CSV
- NGO 代表可看個案使用紀錄、立心可看全域

### Release 2 — 相機掃描 + 體驗優化
- 第二次上傳支援相機掃描流水號 (手機原生 OCR)
- 一次連續掃多張的快速模式
- 立心儀表板（店家用量、NGO 比較）
- 收據自動歸檔

## 6. 已確認的關鍵決定

| # | 決定 |
|---|------|
| 1 | 券單價 NT$100 統一 |
| 2 | NGO 與 NGO 代表合併為一個角色 |
| 3 | 補款由 NGO 代表現場帶現金 |
| 4 | 一個 NGO 代表負責多家店；NGO–店家對應每月固定不輪替 |
| 5 | 未發出券沒月份；發放時才寫月份 → 系統內所有上傳的券都是當月用 |
| 6 | 個案沒帳號、不需身分驗證 |
| 7 | 店家被動，不登入，由 NGO 代表代操 |
| 8 | 流水號由立心發放但無編碼邏輯 |
| 9 | 跨 NGO 用券假設不會發生 |
| 10 | 機構用印處系統不管 |
| 11 | 店家用印處一定會蓋，無容錯需求 |
| 12 | 個案資訊：姓名 + NGO + 身分證字號 (明碼 v1) |
| 13 | 立心月度結算單可下載 CSV |
| 14 | NGO 代表上傳收據；立心可檢視 |
| 15 | 系統不追蹤實體券庫存 |
| 16 | OCR 用手機原生 (iOS Vision / Android ML Kit)，不需 AI API |
| 17 | 個案 + 店家由 NGO 代表自建 |
| 18 | NGO 代表帳號由立心建立 |
| 19 | 金流方向：立心 → NGO 代表 → 店家 (單一鏈) |
| 20 | 立心 = admin + 自家 NGO 代表 雙重身份 (role=lixin 但 ngo_id 指向立心 NGO) |
| 21 | **發佈方式：PWA (漸進式網頁 App)，不上 App Store / Play Store**。使用者用瀏覽器「加入主畫面」即可。Next.js 單一專案，OCR 用 Tesseract.js (browser-side)。省 Apple Dev fee、不用審核、改版即時生效。|

## 7. 不在 v1 範圍

- 推播通知
- 多語言（中文 only）
- OCR 之外的離線模式
- 店家 portal / 個案 App
- 跨 NGO 用券
- 預測 / AI 推薦
- 真實圖像識別店家章
